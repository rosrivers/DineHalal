***Dine Halal - Firebase and Google Sign-In Setup***


This project is for the Dine Halal app, where users can rate halal restaurants and food. 
The primary goal is to integrate Firebase for backend services and Google Sign-In for user authentication. 
Below is an explanation of the setup and configuration done so far.

***Technologies Used***

Firebase: Used for authentication and backend services.
Google Sign-In: Used for user authentication via Google OAuth.
SwiftUI: For the user interface development.
Xcode: Used as the development environment.

1. Firebase Setup
Firebase Authentication: Configured Firebase Authentication to handle user sign-ins. The app is set up to work with Google OAuth for user login.

2. Google Sign-In:

Integrated the Google Sign-In SDK into the app.
Set up the GoogleService-Info.plist file (this file should be kept secure and not pushed to the repository).
Configured Google Sign-In with Firebase Authentication in the app.

3. GoogleService-Info.plist: This file contains the configuration required for Firebase and Google Sign-In to work. Remember, it should not be included in the GitHub repository for security reasons.
4. A .gitignore file is included to ensure the file is not tracked by Git.



***Steps Taken:**
1. Firebase and Google Sign-In Configuration:
The Dine_HalalApp.swift file is configured to initialize Firebase using FirebaseApp.configure().

Google Sign-In is configured using the GIDSignIn.sharedInstance method, passing in the client ID from Firebase's configuration.

2. Handling Sign-In:
The Google sign-in process is not yet implemented.
** This will be handled in a separate view model, and not directly in the ContentView.swift.****

3. Environment Setup:
Followed the proper steps to integrate Firebase into the project and authenticated using Google Sign-In.
These configurations should be tested and verified during the development phase sometime later.


***.gitignore File Explanation.*** 

The .gitignore file is used to specify which files and directories should be ignored by Git. 
This is useful to ensure sensitive data, unnecessary files, or files that are automatically generated during development (like build artifacts or dependency files) do not get committed to the repository.

Contents of .gitignore:
Firebase Configuration Files:
GoogleService-Info.plist: This is the configuration file for Firebase. 
It contains sensitive information such as your Firebase project credentials and API keys, which should not be committed to the repository for security purposes.

Xcode Generated Files:
*.xcuserstate, *.xcworkspace, *.xcodeproj: These are automatically generated by Xcode during development. They store project settings, workspace information, and user-specific data.
DerivedData/: This directory contains cached data from Xcode builds, which is not necessary to be versioned.


CocoaPods: (entirely and totally optional and not part of project). 
Pods/: This directory contains dependencies managed by CocoaPods, and it should not be committed because it can be re-generated by running pod install.
Podfile.lock: Locks the specific versions of the installed pods. It's generally not needed in the repository since dependencies are listed in the Podfile.


API Keys or Secrets:
Secrets.swift: Used for storing any API keys or secrets for your app. This file should be excluded from version control to avoid exposing sensitive information.
macOS System Files:

.DS_Store: A macOS system file used by Finder to store directory settings, which should not be tracked by Git.
